Lexora AI

Document RAG · FastAPI · LangChain · FAISS · OpenAI API · Redis · Celery · PostgreSQL

Lexora AI is a document question-answering platform. Users upload private documents, the backend extracts and chunks text, embeddings are stored in per-user FAISS indexes, and questions are answered with retrieval-augmented generation plus source metadata.

System diagram

2. Parse + chunkDocument parsers extract text and chunk it for embedding.
3. Embed + indexOpenAI embeddings are persisted in a per-user FAISS vector store.
4. Retrieve + cacheQueries search the user index, apply document filters, and cache context in Redis.
5. AnswerLLM response streams back with source metadata and conversation history.

What I built

The repository is organized as a production-style FastAPI backend. Route modules handle authentication, documents, and chat. Service modules handle documents, embeddings, vectors, retrieval, caching, LLM calls, and chat orchestration. PostgreSQL stores users, documents, conversations, and messages; Redis supports token revocation and retrieval caching; Celery is available for background document processing.

The RAG path is intentionally user-scoped. FAISS indexes live under a user-specific path, and retrieval cache keys include user ID, query hash, and document filter. This avoids a common mistake in document AI systems: accidentally mixing one user’s private context into another user’s answer.

Authentication

JWT access and refresh tokens include token IDs. Logout and refresh rotation blacklist used tokens in Redis until expiry.

Document ingestion

The document service validates uploads, extracts text, chunks content, creates embeddings, writes vectors to FAISS, and stores metadata for source attribution.

Retrieval

The retrieval service embeds the query, searches the user vector store, applies document filters inside the search path, deduplicates results by document, and builds compact context for the LLM.

Streaming chat

The chat service supports normal and SSE streaming responses, persists user and assistant messages, passes real chat history to the LLM, and returns source metadata.

Operations

The app exposes health, readiness, and Prometheus metrics endpoints and includes a Docker Compose stack for PostgreSQL, Redis, the API, Celery worker, and Nginx.

Engineering takeaways

  • RAG quality depends heavily on boring systems details: chunking, metadata, filters, user isolation, and cache keys.
  • Source attribution should be part of the data model, not a UI afterthought.
  • Streaming is useful because retrieval and generation can feel slow even when the backend is working correctly.
  • Private document AI needs careful auth, token revocation, CORS configuration, and environment-secret discipline before public deployment.
