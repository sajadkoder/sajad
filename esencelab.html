EsenceLab

Recruitment platform · Next.js · TypeScript · Express API · FastAPI AI service · Supabase/Postgres · Gemini/Groq-style AI workflows

EsenceLab is a campus recruitment platform that turns resume screening into a structured workflow. Students upload resumes and get career guidance, recruiters rank candidates by fit, and admins manage users, recruiter requests, resumes, monitoring, and moderation from one control surface.

What I built

The repository is split into three services: a Next.js frontend, a TypeScript Express backend, and a FastAPI AI service. The backend owns authentication, role authorization, protected routes, recruiter request approval, job/application flows, audit logs, monitoring metrics, and persistence orchestration. The AI service handles PDF parsing, skill extraction, TF-IDF-style matching, and AI-assisted student guidance.

The most important engineering choice was separating product roles clearly. Student flows focus on resume strength, skill gaps, roadmaps, learning plans, jobs, applications, and interview practice. Recruiter flows focus on job posting and candidate ranking. Admin flows focus on moderation, user management, recruiter approval, monitoring, and auditability.

Resume intelligence

The AI service extracts text from PDFs, normalizes skills, splits resume sections, parses education and experience entries, and returns structured data the backend can use for scoring and matching.

Candidate ranking

Recruiters can compare candidates against job requirements using match scores and explanations instead of manually reading every resume from scratch.

Career engine

The backend contains a career engine for resume strength scoring, role-based roadmaps, learning plans, weekly planners, and mock interview packs.

Security and operations

The codebase includes JWT auth, password hashing, Helmet, rate limiting, request IDs, normalized error responses, Supabase-backed persistence, RBAC tests, audit logs, and deployment docs.

Leadership and delivery

I led a 4-engineer team across architecture, sprint planning, implementation, PR reviews, and deployment. The project shipped as a working product rather than a static demo: there are protected user flows, role-specific dashboards, persistent data, production environment templates, Dockerfiles, CI configuration, and runtime verification docs.

Before launch, we also audited the project for hardcoded credentials and unsafe configuration patterns. The README and environment templates make secrets explicit without publishing real credentials.

Engineering takeaways

  • AI product work is mostly integration work: the model output must fit the database, backend permissions, and UI decisions.
  • Role-based software becomes much easier to reason about when student, recruiter, and admin flows are separated early.
  • Security basics matter in college projects too: JWTs, RLS-aware persistence, rate limits, request IDs, and secret hygiene prevent avoidable mistakes.
  • A demo becomes more credible when it has tests, deployment docs, operational notes, and failure-path handling.
