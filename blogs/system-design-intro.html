1. What is System Design

System design is about making architectural decisions when building software that needs to work at scale. It's not about specific technologies—it's about understanding tradeoffs.

When you're building an app for 100 users, almost any approach works. When it's 100 million users, every decision matters. System design is the skill of reasoning through those decisions.

Why It Matters

  • Interviews: FAANG and similar companies test this heavily
  • Real work: Understanding why systems fail helps you build better ones
  • Career growth: Senior engineers are expected to think architecturally
  • Debugging: When things break at scale, you need mental models

System design isn't memorizing architectures. It's learning to ask the right questions and understand tradeoffs.

2. Thinking About Scale

The first question in any system design discussion: "At what scale?"

The Numbers You Should Know

Latency Comparisons

L1 cache: 1ns
RAM: 100ns
SSD seek: 100μs
Disk seek: 10ms
Network (same DC): 0.5ms
Network (cross-region): 150ms

Capacity Estimates

Requests/sec thresholds:
1K: single server
10K: need caching
100K: need load balancing
1M+: distributed systems

Back-of-the-Envelope Math

Quick calculations help sanity-check designs:

  • Storage: 1M users × 1KB data = 1GB. Add 10× for redundancy, indexing.
  • Bandwidth: 1M users × 10 requests/day × 10KB = 100GB/day ≈ 1MB/s
  • Read/Write ratio: Most apps are read-heavy (100:1 or more). Design for it.

Vertical vs Horizontal Scaling

Vertical Scaling (Scale Up)

Bigger machine. More RAM, faster CPU, bigger disk.

Pros: Simple, no code changes. Cons: Expensive, hard limit, single point of failure.

Horizontal Scaling (Scale Out)

More machines. Distribute load across commodity hardware.

Pros: Cheaper, fault tolerant, theoretically unlimited. Cons: Complexity, distributed systems problems.

3. Core Components

Most distributed systems are built from a set of standard components. Understanding each one helps you compose architectures.

Load Balancers

Distribute incoming requests across multiple servers. The entry point to any scaled system.

Load Balancing Algorithms

Round Robin: Rotate through servers sequentially.
Least Connections: Route to server with fewest active requests.
Consistent Hashing: Route same user to same server (sticky sessions).
Weighted: More powerful servers get more traffic.

Tools: Nginx, HAProxy, AWS ALB, Cloudflare
[Users] → [Load Balancer] → [Server 1, Server 2, Server 3] ↓ Health checks remove unhealthy servers from rotation

Reverse Proxy vs API Gateway

Reverse Proxy

Accepts client requests, forwards to backend servers. Handles SSL termination, compression, caching.

Nginx, Apache, HAProxy

API Gateway

Reverse proxy + features. Rate limiting, authentication, request routing, response transformation.

Kong, AWS API Gateway, Ambassador

Message Queues

Decouple services. Producer sends message, consumer processes asynchronously.

When to Use

• Email sending (don't block the request)
• Image processing (heavy work in background)
• Analytics events (batch process later)
• Task scheduling (run at specific times)

RabbitMQ, Kafka, SQS, Redis Streams
[Service A] → [Queue] → [Service B] ↓ ↓ ↓ Produces msg Buffers Consumes & processes Continues asynchronously

4. Database Choices

Database selection is often the most critical architectural decision. Pick wrong and you'll feel it for years.

SQL vs NoSQL

Relational (SQL)

Structured data, relations, ACID transactions, clear schemas.

PostgreSQL, MySQL, SQLite. Use when: data is relational, consistency matters, schema is stable.

Document (NoSQL)

Flexible schemas, nested data, horizontal scaling, eventual consistency.

MongoDB, DynamoDB. Use when: schema evolves, need horizontal scaling, data is hierarchical.

Key-Value

Simple lookups, extremely fast, no complex queries.

Redis, Memcached. Use for: caching, sessions, leaderboards.

Wide-Column

Billions of rows, time-series, high write throughput.

Cassandra, HBase. Use for: logs, metrics, IoT data.

Database Scaling

When one database isn't enough:

Read Replicas

Master handles writes. Replicas handle reads. Works because most apps are read-heavy.

[Writes] → [Master DB] → Replicates → [Replica 1, Replica 2] ↓ [Reads] → Load balancer → [Replica 1, Replica 2]

Sharding (Partitioning)

Split data across multiple databases. Each shard holds a subset of data.

  • Key-based: User ID mod N determines shard
  • Range-based: A-M on shard 1, N-Z on shard 2
  • Directory-based: Lookup table maps keys to shards

The Challenge

Sharding complicates everything: cross-shard queries, transactions, rebalancing. Don't shard until you need to.

5. Caching Strategies

Caching is often the difference between a slow system and a fast one. But cache invalidation is famously "one of the hardest things in CS."

Where to Cache

Client-side (Browser)

HTTP headers (Cache-Control, ETag). Reduces requests entirely.

CDN (Edge)

Static assets, some API responses. Close to users geographically.

Application-level (Redis/Memcached)

Database query results, computed data, sessions.

Database-level

Query cache, buffer pool. Built into most databases.

Cache Patterns

Cache-Aside (Lazy Loading)

1. App checks cache 2. If miss: load from DB, store in cache 3. Return data 4. On write: update DB, invalidate cache

Write-Through

1. App writes to cache 2. Cache writes to DB 3. Read: always from cache (never stale)

Write-Behind (Write-Back)

1. App writes to cache 2. Ack immediately 3. Async: cache writes to DB 4. Risk: data loss if cache crashes before write

Cache Invalidation

  • TTL: Easiest. Expire after X seconds. Stale data for X seconds.
  • Write-invalidate: Delete from cache on write. Next read refreshes.
  • Write-through: Update cache on write. Always fresh, more writes.
  • Event-based: Listen to change events, update cache proactively.

6. Consistency Models

In distributed systems, you can't have everything. The CAP theorem tells us: when partitioned, choose consistency or availability.

CAP Theorem

  • Consistency: All nodes see the same data at the same time
  • Availability: Every request gets a response (success or failure)
  • Partition Tolerance: System continues despite network failures

Network partitions are inevitable in distributed systems. So you're really choosing between CP and AP.

Consistency Levels

Strong Consistency

All reads see the most recent write. Like a single-threaded program.

Trade-off: Higher latency, lower availability. Banking systems need this.

Eventual Consistency

Given enough time, all replicas converge. Reads might see stale data temporarily.

Trade-off: Lower latency, higher availability. Social media feeds can tolerate this.

Causal Consistency

Events that are causally related are seen by everyone in order.

If user A replies to user B's comment, everyone sees B's comment before A's reply.

There's no "best" consistency model. The right choice depends on your application's requirements.

7. Common Patterns

Certain patterns show up repeatedly in system design. Learn these and you'll recognize them everywhere.

Microservices

Break monolith into small, independent services. Each owns its data and logic.

  • Pros: Independent deployment, technology flexibility, team autonomy
  • Cons: Distributed complexity, network overhead, harder debugging

Don't start with microservices. Extract from monolith when boundaries become clear.

Event-Driven Architecture

Services communicate via events, not direct calls. Looser coupling.

[Order Service] → "OrderCreated" event → [Inventory Service] ↓ [Email Service] ↓ [Analytics Service] All services react independently. No direct coupling.

CQRS (Command Query Responsibility Segregation)

Separate models for reading and writing data. Optimize each independently.

  • Write model: Normalized, transactional, authoritative
  • Read model: Denormalized, optimized for queries, can be stale

Useful when read and write patterns differ significantly.

Circuit Breaker

Prevent cascade failures. If a service is failing, stop calling it.

States: CLOSED → OPEN → HALF-OPEN → CLOSED ↓ ↓ ↓ Normal Failing Testing if recovered requests requests (limited calls) pass fail fast

Rate Limiting

Protect services from overload. Common algorithms:

  • Token Bucket: Refill tokens over time, deny when empty
  • Sliding Window: Count requests in last N seconds
  • Fixed Window: Reset count every N seconds

8. The Design Process

When faced with a system design problem, follow a structured approach:

Step 1: Clarify Requirements

  • What's the expected scale? (users, requests/sec, data size)
  • What are the functional requirements? (features)
  • What are the non-functional requirements? (latency, availability)
  • What are the constraints? (budget, existing infrastructure)

Step 2: Define Data Model

  • What entities exist?
  • How do they relate?
  • What queries do we need?
  • How much data? How fast is it growing?

Step 3: Design APIs

  • What endpoints/operations?
  • Request/response formats?
  • Authentication/authorization?

Step 4: High-Level Design

  • Draw the major components
  • Show data flow
  • Identify bottlenecks

Step 5: Deep Dive

  • How does each component work?
  • How do they handle failure?
  • Where can we optimize?

Step 6: Scale & Iterate

  • Add caching where needed
  • Introduce load balancing
  • Consider sharding
  • Add monitoring

9. How to Learn

System design is learned through exposure and practice. Here's my approach:

Resources That Helped Me

  • Designing Data-Intensive Applications: The bible. Read it, re-read it.
  • System Design Primer (GitHub): Free, comprehensive, practical
  • ByteByteGo: Good visual explanations of concepts
  • Real architecture blogs: Uber, Netflix, Discord engineering blogs

Practice Approach

  • Design something from scratch (URL shortener, chat app, feed)
  • Compare your design to real systems
  • Understand why they made different choices
  • Present designs to others—explaining reveals gaps

The Reality

You won't design distributed systems from scratch early in your career. But understanding the concepts helps you:

  • Debug issues in existing systems
  • Make better decisions at any scale
  • Understand why things are built certain ways
  • Communicate with senior engineers effectively

System design isn't about knowing the "right" answer. It's about reasoning through tradeoffs and asking the right questions.
