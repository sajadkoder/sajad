SpeakSwap

Browser app · speech input/output · translation fallbacks

SpeakSwap began as a college mini-project. I wanted to build a translation tool that felt simple to use: type or speak, get the translation, and hear it spoken back.

How it works

The client handles the interface, speech recognition, and speech synthesis using browser APIs. Translation requests go through a server-side layer with fallback services, rate limiting, and caching so the app can fail more gracefully when one service is unavailable.

What made it interesting

The difficult parts were not the happy path. Browser speech APIs behave differently across environments, translation services can fail or rate-limit, and voice input needs an explicit language before listening. Those constraints made the project a good exercise in practical edge-case handling.

What I learned

  • Small user-facing apps still need careful fallback design.
  • Speech interfaces require more browser-specific testing than normal text inputs.
  • Keeping the UI simple matters when the backend services are uncertain.
