SentinelML

Fraud detection · FastAPI · scikit-learn · PyTorch experiments · Redis · PostgreSQL · MLflow · Docker

SentinelML is a fraud detection service built around the full machine-learning system loop: generate or ingest transactions, engineer fraud signals, train and evaluate models, serve predictions through an API, cache repeated requests, log predictions, and monitor model behavior after deployment.

What I built

The codebase is structured like a deployable ML service, not just a notebook. sentinel_ml/api/main.py defines request/response schemas, the prediction service, health checks, batch prediction, model metadata, dashboard routes, Redis caching, and optional database logging. The training code wraps model creation, metrics, cross-validation, artifact saving, and MLflow integration.

The public repository currently focuses on a robust scikit-learn service path with Random Forest, Gradient Boosting, and Logistic Regression support. My resume summarizes the strongest fraud-detection experiment: engineered features, imbalanced-data handling, and an ensemble-style run that reached 0.94 ROC-AUC on held-out data.

Feature engineering

The feature layer builds transaction-level signals from amount, merchant category, transaction type, country, device type, time, and user behavior. The feature engineer can be saved and loaded with Joblib for consistent training and inference.

Prediction service

The API accepts single transactions and batches, computes a fraud score, applies a configurable threshold, returns confidence labels, and includes model version plus inference latency in the response.

Caching and auditability

Redis caches deterministic prediction requests by payload and model version. PostgreSQL logging is optional, so local predictions continue even when the database is unavailable.

Monitoring

The repository includes drift/performance utilities, health endpoints, dashboard assets, Prometheus config, Grafana service wiring, and a load-test script for checking API behavior under traffic.

Engineering takeaways

  • Model accuracy is only useful if the serving path is stable, observable, and reproducible.
  • Caching should include the model version, otherwise old predictions can leak into new model behavior.
  • Optional infrastructure makes local development much easier: Redis and PostgreSQL improve the system, but predictions should still work without them.
  • Fraud models need more than a score. A useful service also returns confidence, version, latency, and audit metadata.
