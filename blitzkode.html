BlitzKode

Coding assistant · Qwen2.5-Coder · SFT/GRPO/DPO · LoRA · GGUF · FastAPI · llama.cpp

BlitzKode is my local-first coding assistant project. The goal was to understand the full engineering path behind a useful coding model: prepare data, fine-tune a Qwen2.5-Coder checkpoint, reduce training memory with LoRA, export the result to GGUF, and serve it through a small FastAPI runtime backed by llama.cpp.

What I built

The public repository is intentionally API-first. The central file is server.py, which builds the FastAPI app, validates requests, lazily loads the local GGUF model, formats prompts, serializes model access with an async lock, and exposes JSON and Server-Sent Events generation endpoints.

The repo also includes training and operator scripts for dataset building, LoRA SFT, DPO, smoke inference, GGUF export, Hugging Face publishing, and local evaluation. This made the project more than a single notebook: it became a reproducible model-development and serving workflow.

Inference path

/generate handles normal JSON generation. /generate/stream runs llama.cpp in a worker thread and bridges tokens into SSE messages for responsive clients.

Research mode

/search/web and /generate/research add DuckDuckGo-backed search snippets as untrusted context. The prompt explicitly tells the model not to invent APIs, citations, files, or execution results.

Runtime controls

Environment variables tune context length, CPU threads, batch size, micro-batch size, GPU layer offload, mmap loading, prompt cache behavior, bearer auth, CORS, rate limiting, and request-size limits.

Guardrails

The server includes a grounding guard for direct unknown API/signature questions. This exists because small local models can still hallucinate plausible function signatures without source context.

Training and evaluation

The resume version summarizes the larger Qwen2.5-Coder training direction. The repository also keeps earlier Qwen2.5-family artifacts and evaluation notes, which helped me iterate on the pipeline before publishing. The core training idea stayed the same: use supervised fine-tuning for baseline coding behavior, preference optimization to reduce weak answers, and GGUF export to make the result usable locally.

The included evaluation harness checks practical coding prompts such as factorial, binary search, SQL aggregation, and uncertainty around fictional APIs. One logged GGUF smoke run passed 3 out of 4 cases: it handled Python and SQL generation, but still hallucinated an imaginary API signature. That failure shaped the server-side guardrail and the plan for stronger DPO examples.

Engineering takeaways

  • Fine-tuning is only one part of the product. Dataset quality, evaluation, packaging, and runtime behavior matter just as much.
  • Local inference changes the UX: quantization, model loading, token streaming, CPU threads, and context length directly affect whether the assistant feels usable.
  • Small models need honest guardrails. If a model is likely to invent an API, the system should detect that pattern instead of pretending the model is reliable.
  • A model project is easier to maintain when training, export, evaluation, serving, and deployment are all represented in the repo.
